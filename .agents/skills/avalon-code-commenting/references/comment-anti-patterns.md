# 注释反模式

## 1. 机械翻译代码动作

以下写法默认不合格：

- `// 这里查询数据库`
- `// 这里遍历列表`
- `// 这里返回结果`

这些内容从代码本身就能看出来，没有提供额外信息。

接口方法如果只写下面这种 KDoc，也视为低信息量：

- `/** 获取用户 */`
- `/** 保存数据 @param id id @return 结果 */`

KDoc 不是把签名翻译一遍，而是补充调用方真正需要知道的契约、约束和设计意图。

## 2. 只写大而空的文件头

如果文件头只写“这是用户服务”“这是仓储实现”，但没有解释边界、取舍或关键约束，价值很低。

模块入口注释不能替代核心方法上的就地说明。

## 3. 用注释掩盖坏命名

如果变量、方法、类型名本身含糊，不要靠长注释补救。先改清楚命名，再补必要说明。

## 4. 注释比实现更旧

以下情况必须处理，而不是保留：

- 注释还在描述旧事务边界
- 注释还在描述已经删掉的字段或流程
- 注释里的“当前阶段”已经和现状不一致

遇到这类情况，优先更新或删除旧注释。

## 5. 局部行内注释过密

不要给每一行都加中文解释。真正需要的是：

- 方法前说明整体数据流
- 在局部难点处补一两句关键提示

同理，也不要为了“全部 KDoc 化”给每个简单 getter / setter / DTO 字段都写无信息量 KDoc。

但对外边界 DTO 不在这个豁免范围内。问题不在于“要不要写”，而在于不要把字段说明写成：

- `@property code 编码`
- `@property message 消息`

这种写法仍然没有告诉调用方字段的真实业务语义。

## 6. 重复搬运仓库外说明

不要把一大段外部文档原封不动搬进代码注释。代码注释只保留当前文件理解必须依赖的信息。

## 7. 计划型注释长期滞留

像下面这类注释很容易过期：

- `TODO: 第二阶段再做`
- `后面也许会支持更多模式`
- `未来准备拆出去`

除非它直接影响当前代码理解或明确是近期执行入口，否则不要长期保留在正式实现里。
