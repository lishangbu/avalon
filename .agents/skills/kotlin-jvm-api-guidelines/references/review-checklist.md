# Kotlin/JVM 评审清单

提交或评审前至少检查：

1. 这段代码是内部实现还是公共 API，是否按对应标准设计
2. 公共 API 是否足够显式，尤其是 visibility、返回类型、nullability
3. 命名、参数顺序、重载和错误处理是否与同组 API 保持一致
4. 是否复用了 Kotlin 现有概念，而不是发明近似抽象
5. 是否避免了不必要的布尔参数、可变集合暴露和数组泄漏
6. 顶层函数、成员函数、扩展函数、DSL 的选型是否清晰合理
7. 文件组织是否按概念阅读优化，而不是把不相关声明塞进泛名文件
8. 当前文件里的多个声明是否语义上紧密相关，且大小仍然合理
9. 是否已经出现应拆分的信号，例如独立调用方、独立 KDoc、文件名泛化
10. 如果明确存在 Java 调用方，这个 API 的 `@Jvm*` 设计是否明确且克制；否则不要额外为了 Java 改写 Kotlin-first 设计
11. 是否存在明显的稳定性或兼容性风险，例如签名漂移、默认参数滥用、层级随意开放
12. 公共 API 的 KDoc 是否用准确写法说明了真实契约，而不是重复签名；仓库级强制范围是否交由 `avalon-code-commenting` 处理
13. 公开承诺的行为是否有对应测试，尤其是边界条件和失败路径
