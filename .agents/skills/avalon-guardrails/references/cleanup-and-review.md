# 清理与评审规则

## 1. 清理规则

### 1.1 只清理当前改动直接相关的范围

- 不要在做一个功能时，顺手大面积清理无关模块、无关包、无关命名
- 清理默认限制在当前改动覆盖的上下文、模块和调用链内

### 1.2 占位文件要么有职责，要么删除

- 像空的 `*Module.kt`、空配置类、空适配器、空工具类这类文件，如果没有明确职责，不要长期保留
- 有明确职责就补全职责
- 没有明确职责就删除

### 1.3 重复实现优先收敛

- 如果发现重复 DTO、重复 SQL、重复映射、重复异常包装、重复事务编排，优先收敛
- 不保留“未来可能会用”的近似副本
- 不要为了暂时省事保留两个语义几乎相同的实现

### 1.4 死代码及时清理

包括但不限于：

- 未使用的类、方法、字段、常量
- 已废弃但无人引用的适配层
- 不再使用的配置项
- 已失效的测试夹具和假数据

### 1.5 清理不能破坏边界

- 清理不能把上下文专属代码随意上移到 `shared-*`
- 清理不能为了省事把跨上下文逻辑揉在一起
- 清理应让模块边界更清楚，而不是更模糊

### 1.6 Flyway migration 不做回写式清理

- 已进入仓库并可能被执行过的 migration，不做改旧文件式清理
- 表结构调整通过新增 migration 完成
- 只有在明确未使用且任务上下文允许时，才考虑修改最新 migration

### 1.7 配置清理保持保守

删除配置前先确认：

- 当前代码已无引用
- 各 profile 中已无引用
- 启动链路和现有代码说明不依赖它
- 它不是近期明确会启用的预留配置

### 1.8 测试和代码说明随清理一起更新

如果清理影响了：

- 路径
- 类型
- SQL
- 响应结构
- 启动配置

则对应测试、模块入口说明、核心产物注释和技能 reference 也要同步更新。

### 1.9 不因为美观做高风险清理

- 批量改名、跨模块搬运、统一重排目录这类高扰动清理，不要作为顺手操作混入普通任务
- 如果清理已经明显超出当前任务主目标，应单独作为任务处理

### 1.10 清理后必须验证

- 只要发生清理，就至少执行与范围匹配的最小验证
- 默认优先做编译、受影响模块测试，以及必要的集成测试

## 2. 通用评审入口

在提交任何后端改动前，至少检查：

1. 这次改动的主上下文是否单一明确
2. 是否新增了不该存在的 shared 依赖
3. 是否突破了 schema 隔离
4. 是否出现跨上下文直连
5. 是否存在写库加发消息的裸双写
6. 是否应当补 outbox
7. 安全接入是否仍遵循当前认证基线
8. 测试策略是否与改动层次匹配，且覆盖了关键行为、事务边界或集成边界
9. 新引入的依赖、序列化方式或底层集成是否已评估 `native` 支持与风险
10. 当前改动涉及的 `deprecated` API 是否已被替换，或已明确记录未替换原因

如果本次改动还触碰了以下专题，继续转到对应专项清单：

1. 涉及模块入口、接口方法 KDoc、边界 DTO、复杂单元注释时，继续看
   `avalon-code-commenting/references/comment-review-checklist.md`
2. 涉及 Kotlin/JVM 公共 API、文件组织、`@Jvm*`、兼容性或 KDoc 写法时，继续看
   `kotlin-jvm-api-guidelines/references/review-checklist.md`
