# 幂等与反模式

## 1. 消费端幂等

所有消费 outbox 发布事件的下游，都应该具备处理重复事件的策略。

常见做法：

- 存储已处理的 `event_id`
- 通过业务唯一键让写入天然幂等
- 把重复更新设计成无害的 `upsert`

## 2. 重试规则

- 显式记录重试次数
- 显式保留最后一次错误
- 失败后退避，不要热循环重试

## 3. 必须拒绝的模式

- 先写业务数据，再直接发布，然后寄希望于两者都成功
- 在事务里直接发布，并假设回滚能撤销外部副作用
- 让下游依赖 at-most-once 交付
- 直接把持久化行模型序列化成事件契约

## 4. 评审清单

1. 同一条事件被处理两次时，是否仍然安全
2. 发布失败后重试，是否不会破坏状态
3. 事件负载是否可版本化且语义明确
4. 是否还残留隐藏式双写
