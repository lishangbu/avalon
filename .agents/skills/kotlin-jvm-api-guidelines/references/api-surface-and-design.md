# 公共 API 设计

## 1. 降低 mental complexity

- 调用方应该容易猜到函数做什么、返回什么、失败时会怎样。
- 同一组 API 的命名、参数顺序、重载行为和错误处理机制要保持一致。
- 如果一个新入口会让调用方难以判断它和已有入口的差别，先考虑是否真的需要新增。

## 2. 参数顺序与命名

- 参数从通用到具体，从必填到可选。
- 同一概念在整个 API 中只用一种术语，不要一会儿 `startIndex` 一会儿 `beginIndex`。
- 同语义不同输入类型的重载，行为应尽量一致。
- 多个同类型参数如果容易混淆，应重新建模，而不是逼调用方硬记顺序。

## 3. 默认值与“默认做对的事”

- 公共 API 应尽量让常见用法不需要额外传默认值。
- 如果 happy path 不明显，先重审需求和抽象，而不是继续加参数。
- 对可能持续扩展的配置，优先考虑配置对象或 DSL，而不是不断扩充长参数列表。

## 4. 状态与扩展性

- 数据与状态优先建模为类型，而不是散落在函数参数里。
- 当层级是封闭的，优先用 sealed 限制非法扩展。
- 当你确实希望用户扩展时，要让扩展点显式而清楚，不要让“任意继承”成为默认行为。

## 5. 可变状态与集合

- 公共 API 默认优先使用只读集合。
- 不要把内部可变集合直接暴露出去。
- 如果必须处理数组或可变结构，考虑 defensive copy，并清楚界定所有权。

## 6. 输入校验

- 输入前置条件优先用 `require`。
- 实例或流程状态校验优先用 `check`。
- 错误消息要能帮助定位问题，但不要泄露敏感信息。

## 7. 测试与质量

- 所有公开承诺过的行为都应该被测试覆盖。
- 需要特别覆盖边界值、空输入、非法输入和兼容性敏感路径。
- 静态检查、格式化、测试和覆盖率报告应成为公共 API 演进的基础保障。
