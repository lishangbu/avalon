# 文件组织与聚合规则

## 1. 默认原则

Kotlin 默认应偏向“一份文件一个公开顶层概念”。

这里的“概念”不只是 `class` 关键字，也包括：

- 公共 `interface`
- 公共 `class`
- 公共 `object`
- 公共 `enum class`
- 公共 `value class`
- 会被跨包、跨模块直接依赖的领域模型
- 应用服务、仓储、资源、适配器这类入口类型

如果一个声明有独立职责、独立 KDoc、独立调用方和独立演进节奏，通常就值得单独成文件。

在 `avalon` 仓库里，默认进一步收紧为：

- 每个公开顶层 `class`、`interface`、`object`、`enum class`、`value class` 各自单独成文件
- `companion object` 作为宿主类型内部实现的一部分，不单独计算为额外概念
- 只有 `private` / `internal` helper、与宿主类型强绑定的少量扩展函数或工厂函数，才允许与主类型同文件

## 2. Kotlin 官方基线

Kotlin coding conventions 明确给出两条重要基线：

- 如果一个 Kotlin 文件只包含一个 class 或 interface（允许附带紧邻的相关顶层声明），文件名应与该类型同名。
- 允许在同一文件中放多个声明，但前提是这些声明在语义上紧密相关，且文件大小保持合理。

这意味着：

- Kotlin 并不要求机械地“一类一文件”
- 但也不鼓励把不相关声明为了省事堆在一起

## 3. 什么情况应该单独成文件

以下情况默认拆分：

- 一个公开类型本身就是模块或调用链入口
- 类型会被多个文件、多个包或多个模块直接引用
- 类型需要独立 KDoc 承载完整契约
- 类型未来很可能独立演进
- 文件名如果继续聚合只能退化成 `Models`、`Dtos`、`Utils` 这类泛名
- 聚合后阅读者需要在一个文件里跳来跳去才能理解多个不相关概念

典型例子：

- `IdentityAccessService.kt`
- `AuthenticationRepository.kt`
- `TypeDefinitionResource.kt`
- `MenuType.kt`

## 4. 什么情况允许聚合

只有在“强内聚、同生命周期、一起读更清楚”时，才推荐聚合。

以下情况通常可以放在同一文件：

- 同一个小型 sealed 家族
- 同一个小概念下的一组轻量 `value class`
- 同一个小概念下的一组异常类型
- 只服务一个文件的 `private` / `internal` helper
- 与某个核心类型紧密相关、且面向所有调用方的扩展函数
- 只服务单个公开类型的少量映射函数或工厂函数

判断标准不是“它们都很短”，而是：

- 是不是同一个小主题
- 调用方是不是基本一致
- 修改节奏是不是基本一致
- 放一起是否明显更利于理解

## 5. 哪些聚合是高风险信号

出现以下信号时，应优先考虑拆分：

- 文件中有多个公开类型，但并不围绕同一个小概念
- 每个类型都需要单独一大段 KDoc
- 改一个类型时，经常无关地碰到其他类型
- 文件名已经变成 `Models.kt`、`Dtos.kt`、`Utils.kt`、`Common.kt`
- 同一个文件既有领域模型、又有 DTO、又有 mapper
- 文件长度和注释密度已经让导航成本明显上升

## 6. 对不同类型的建议

### 6.1 领域模型

- 核心聚合、核心实体、重要枚举，默认单独成文件
- 小范围 value class / exception 家族可暂时聚合
- 一旦这些聚合文件中的每个类型都需要独立解释，就应拆分

### 6.2 DTO

- 在 `avalon` 中，`Request`、`Response` 默认各自独立成文件
- 跨多个资源共享的边界 DTO，优先按概念拆分
- 默认不要继续使用 `Dtos.kt` 承载多个公开 DTO；只有在迁移中的过渡阶段才暂时接受

### 6.3 扩展函数

- 对所有调用方都成立的扩展函数，可与核心类型同文件
- 只服务特定调用方的扩展函数，应放在调用方附近
- 不要创建“某个类型的所有扩展函数全集”文件

### 6.4 映射函数

- `Request -> Draft/Command` 一类映射，优先贴近 `Request` 类型放置
- `Domain/Projection -> Response` 一类映射，优先贴近 `Response` 类型放置
- 如果映射只服务一个公开类型，优先与该类型同文件；其次放在同包下的 `*Mappings.kt`
- 只有当同一能力下映射函数数量明显增多、且已不再服务单一类型时，才拆成独立 `*Mappings.kt`
- 不要为了“统一收口”默认创建覆盖整包的 `Dtos.kt`、`Mapper.kt`

## 7. 结合本仓库的落地建议

在 `avalon` 里，默认建议是：

- 资源、服务、仓储、适配器：单文件
- 核心领域模型：优先单文件
- 公开顶层 DTO：一个类型一个文件
- `Ids.kt`、`Exceptions.kt` 这类轻量概念簇：当前可以接受，但要控制规模
- `Models.kt`、`Dtos.kt`：只可作为过渡形态，不应长期成为复杂上下文的主组织方式

## 8. 迁移策略

发现一个聚合文件已经过重时，不要一次性把所有内容炸开重排。更稳的做法是：

1. 先从最常被单独引用的公开类型拆起
2. 再拆拥有独立 KDoc 和独立演进节奏的类型
3. 最后收掉只剩一个含糊主题的大杂烩文件

拆分时优先保证：

- 文件名能表达概念
- 导入路径变化最小
- 同一轮改动中不要顺手大规模重排无关文件
