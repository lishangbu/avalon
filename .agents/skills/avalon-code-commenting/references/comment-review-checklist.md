# 注释评审清单

提交前至少检查：

1. 这次改动是否触碰了“必须补注释”的单元
2. 声明级注释是否默认使用了 KDoc 风格 `/** ... */`
3. 接口、抽象契约和公共方法是否逐个方法补了 KDoc，而不是只写类型头说明
4. 方法 KDoc 是否说明了作用与关键设计思路，而不是只翻译方法名
5. 每个非显然参数是否都有 `@param`
6. 非 `Unit` 返回值是否已有 `@return`，并说明空值 / 空集合 / 失败语义
7. 对外 Request/Response 或其他边界 DTO 是否已有类级 KDoc
8. 边界 DTO 的字段是否通过 `@property` 说明了作用、可空语义、默认行为或格式约束
9. 如果方法有副作用、异常语义、顺序要求或调用约束，是否已经说明
10. 注释是否放在最贴近理解成本的位置，而不是只堆在文件头
11. 注释是否解释了代码本身看不出来的设计原因、边界或取舍
12. 注释是否仍和当前实现一致，没有描述已删除或已变更的流程
13. 是否避免了逐行动作翻译和低信息量废话
14. 模块入口注释是否说明了职责边界和不负责什么
15. 公共抽象注释是否说明了为什么放在这里，而不是放在某个具体上下文
16. 复杂仓储或应用服务注释是否能帮助后来者快速理解编排方式
17. 如果删掉这段注释，是否会丢失关键信息；如果不会，考虑删掉或重写
