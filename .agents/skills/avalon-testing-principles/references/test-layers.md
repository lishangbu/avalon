# 测试层级对照

## 快速判定

| 层级             | 关注点                  | 典型对象                               | 典型断言                |
|----------------|----------------------|------------------------------------|---------------------|
| domain         | 纯业务不变量、状态迁移、值对象      | 聚合、实体、判定函数                         | 结果、约束、边界值           |
| application    | 编排、事务、幂等、outbox、权限   | service、command handler            | 调用顺序、副作用、提交结果       |
| infrastructure | SQL、持久化、调度、适配器       | repository、writer、scheduler、hasher | 真实落库、映射、约束、时序       |
| interfaces     | API 协议、校验、状态码、错误转换   | resource、controller、mapper         | 请求/响应、HTTP code、错误体 |
| integration    | 真实运行路径、跨层回归、少量 smoke | 端到端主流程                             | 关键故事、系统联通性          |

## 选择顺序

1. 能否在内存里完全验证规则。
2. 如果不行，是否只需要应用层协作。
3. 如果需要真实数据库、事务或调度，就上基础设施级验证。
4. 如果只关心 API 契约，再保留接口级测试。
5. 如果问题跨层或风险很高，再补一条集成级主路径。

## 经验法则

- 越靠近纯规则，测试越细。
- 越靠近外部世界，测试越少但越真实。
- 一条测试尽量只覆盖一个层级的主要责任。
- `avalon-app` 里的 HTTP 测试更适合作为 smoke/关键流程验证，不适合作为全部业务规则的唯一防线。

## 覆盖率使用方式

- 把覆盖率当作报警器，不要当作真理。
- 对新功能和缺陷修复，优先看变更范围的测试是否齐全，而不是看整个仓库的平均值。
- 如果要引入覆盖率阈值，优先用它约束关键模块和关键路径。
- 对纯 DTO、简单映射和框架生成的样板代码，不要用覆盖率倒逼无意义测试。

## 严格落地标准

- 新增行为必须有测试。
- 缺陷修复必须有回归测试。
- 关键模块改动默认至少有一条低层测试和一条边界测试。
- `domain` 和 `application` 的规则不能只依赖 `HTTP` 测试兜底。
- `interfaces` 的变动必须验证协议结果，不只验证状态码是否为 200。
- `infrastructure` 的变动尽量使用真实数据库、事务、调度器或适配器验证。
- 如果测试很难写，先检查是不是职责切分太粗，而不是直接降低测试要求。
