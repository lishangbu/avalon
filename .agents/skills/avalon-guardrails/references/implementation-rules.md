# 工程实现规则

## 1. 先理解，再改动

开始写代码前，至少先完成这些动作：

- 定位现有实现和主归属 `bounded context`
- 梳理调用链、事务边界、数据流和事件流
- 搜索相似模块、相似 SQL、相似 DTO、相似 migration、相似异常处理
- 确认仓库里是否已经存在可复用的常量、映射、查询模型、仓储模式和测试模式

不要在没确认已有能力前，直接新写一套相似实现。

### 1.1 重要改动先明确代码内说明承载点

- 重要改动在实现前，先确认哪些模块入口文件和核心产物注释负责说明边界、关键流程和取舍
- 重要改动包括但不限于：
    - 新 `bounded context`、新子域或明显的边界调整
    - 新 `schema`、新核心表或关键索引策略调整
    - 认证与授权方案变更
    - 事务边界、`outbox`、事件模型或跨上下文协作方式变更
    - 会显著影响性能基线或 `native` 基线的底层选型
- 默认通过模块入口文件和核心产物注释承载这些说明

## 2. 先复用，再扩展，再新增

如果出现以下情况，优先考虑复用与抽象：

- 同一段逻辑可能在两处及以上使用
- 准备复制粘贴已有代码
- 同一业务规则、参数拼装、DTO 映射、状态转换在多个位置重复出现
- 同一类 SQL 查询或相同访问路径在不同文件重复实现

处理顺序固定为：

1. 优先复用现有模块
2. 现有模块不完全合适时，优先扩展现有模块
3. 仍不合适，再新增可复用单元

## 3. 先识别热路径，再实现

以下场景默认把性能当作验收项，而不是附加项：

- 高频接口、分页列表、聚合查询
- 批量导入导出、批处理、定时任务
- `outbox` 轮询、事件分发、实时推送
- 大对象序列化 / 反序列化
- 多表关联、递归树查询、复杂过滤与排序
- Redis / 数据库 / 外部服务的高频往返

优先关注：

- 接口延迟
- SQL 次数和数据库往返次数
- 查询复杂度与索引命中
- 序列化成本
- 内存占用
- 事务范围大小
- 异步链路中的重试与重复投递成本

在性能优化时还要同时注意：

- 不要为了局部性能收益默认引入会明显破坏 `native` 可达性的方案
- 如果某个方案依赖重反射、动态代理或非常规运行时技巧，必须单独论证收益与代价

## 4. 不做无关兼容

除非用户明确要求，否则不要主动加入：

- 旧接口兼容分支
- 旧字段兼容分支
- 多余的兜底参数
- “以防万一”的 hack 式容错
- 吞异常式 `try/catch`

## 4.1 过期 API 处理规则

- 当前改动一旦触及 `deprecated` API，默认优先替换为当前依赖基线支持的非过期用法
- 不要因为“现在还能用”就继续扩散新的过期用法
- 如果替换成本明显超出当前任务范围，至少应做到：
    - 不再新增新的过期用法
    - 在结果说明中明确未替换原因
    - 说明影响范围和后续收口入口
- 对底层客户端、框架集成、序列化、认证安全链路中的 `deprecated` API，应提高替换优先级

## 5. 错误处理保持单一出口

- 优先复用统一异常类型和异常映射
- 不要在 `resource`、`application`、`repository` 多层重复包装同一错误
- 不要同时返回业务错误响应又在其他层重复处理同一失败

## 6. 避免明显可预防的性能浪费

以下方向默认不合格：

- 同一请求链路对同一批数据反复查询、反复转换
- 本可一次遍历完成，却拆成多次遍历
- 明明可以通过索引、映射、批量 SQL、集合预聚合优化，却仍保留重复线性扫描
- 在热路径上做明显可前置的计算
- 无必要的全量刷新、全量重算、全量重放
- 事务中夹带不必要的远程调用、重计算或大对象组装

## 7. 类型与契约必须清晰

- 输入、输出、命令、查询、事件、响应 DTO 必须清晰明确
- 优先复用现有类型，不要重复定义近似类型
- 不要让稳定结构长期停留在松散的 `Map<String, Any>` 或匿名 JSON 拼装里

## 8. 注释规范

- 详细范围、模板、反模式和评审清单，以 `avalon-code-commenting` 技能为准
- 注释默认使用中文，专有名词、协议名、框架名、代码标识和数据库对象名保留英文
- 声明级注释默认使用 KDoc 风格 `/** ... */`
- 注释优先解释“为什么这样设计”，以及事务边界、数据流、事件流、边界条件和性能取舍
- 模块级长期说明优先贴近代码提交，默认写在模块入口文件和核心产物的中文块注释中
- 不要把注释写成代码字面动作翻译
- 新增或修改了“必须写注释”的单元，却没有同步补充或更新说明时，默认视为实现未完成

## 9. 测试驱动策略

- 项目默认采用测试驱动开发，但按层执行，不要求所有代码都机械地从接口层先写测试
- 测试优先围绕业务行为、边界条件、事务边界、事件产出和集成契约，而不是框架调用细节
- 领域层优先使用纯单元测试驱动聚合、值对象、领域规则和领域事件
- 应用层优先验证用例编排、权限判断、事务边界、仓储协作和 `outbox` 写入
- 基础设施层优先使用真实 PostgreSQL、Redis 与 Quarkus 运行时的集成测试验证 SQL、迁移、调度和配置
- 事件驱动链路优先验证业务事件契约、`outbox` 状态迁移、分发结果和消费幂等，而不是只校验内部 publisher 是否被调用
- 简单 DTO、直白映射和低价值样板代码不强制机械执行先测后写，但一旦进入核心规则、事务或集成边界，就应优先补齐测试
- 避免用大量 mock 替代真实边界验证；只有在能显著缩小测试范围且不损失行为信心时才使用 mock

## 10. Native 验收策略

- 日常开发、调试和大多数测试仍以 JVM 模式为主
- `native` 支持作为阶段性验收目标，而不是要求每次小改动都立即跑完整 `native build`
- 至少在这些节点优先安排 `native` 构建验证：
    - 基础骨架稳定后
    - 认证链路或关键基础设施链路打通后
    - 每个主上下文首个可用版本完成后
    - 准备发布前
- 如果一次改动引入了新的基础设施依赖、序列化策略、底层客户端或运行时集成，应提高 `native` 验证优先级

## 11. 改动后做最小必要验证

只要发生代码修改，结束前应尽量执行与改动匹配的最小验证，并明确说明：

- 运行了什么验证
- 是否通过
- 若未通过，失败点在哪里

默认优先：

- 编译校验
- 受影响模块测试
- 与当前改动直接相关的集成测试
