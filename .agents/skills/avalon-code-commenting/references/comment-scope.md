# 注释范围与触发规则

## 1. 先判断是否属于必须补注释的单元

以下单元默认必须有中文说明：

- 模块入口文件
- 接口、抽象类、仓储契约、应用服务契约、跨上下文公共服务入口
- 对外 Request/Response、事件 payload、快照响应和其他会被外部调用方直接感知的边界 DTO
- `shared-kernel` 下对外公开的核心抽象、异常基类、公共领域基线类型
- `shared-application` 下对外公开的应用层公共契约、端口和值对象
- 会被多个上下文直接依赖的公共契约、快照类型、ACL 边界类型
- 复杂 `application service`
- 复杂 SQL-first repository
- `outbox writer`、`dispatcher`、幂等处理
- ACL、权限装配、授权快照
- 递归树构建、批处理、投影更新
- 认证、会话、令牌、权限校验等安全敏感路径
- 有明确性能取舍的热点路径

## 2. 什么算“复杂单元”

满足以下任一情况，通常按复杂单元处理：

- 一个方法或类型承担多步业务编排
- 涉及多个仓储、多个表、多个查询阶段或批量更新
- 事务边界、锁、冲突处理、幂等处理不直观
- 需要解释安全语义、会话策略、令牌策略或权限装配逻辑
- 需要解释递归、树结构、聚合装配、快照构建或投影更新
- 存在明显的性能取舍，例如一次查询装配、批量 SQL、避免重复往返

## 3. 什么情况下默认不强制补注释

以下情况优先保持代码直接清晰，不机械补注释：

- 简单 DTO
- 直白字段映射
- 普通 CRUD
- 单步校验且命名已经足够清晰的简单逻辑
- 只改常量值、文案、返回码映射等不会改变设计理解的微小改动

这里的“简单 DTO”不包括对外边界 DTO。只要调用方会直接依赖它的字段语义，就应补 KDoc。

## 4. 注释放置位置

- 模块整体职责、边界和落地顺序：写在模块入口文件的 KDoc 风格总览注释
- 某个公共类型或核心类型的职责与边界：写在类型头部 KDoc
- 接口和抽象契约的方法：逐个方法写 KDoc，说明作用、关键设计思路、`@param`、`@return` 和必要时的副作用 / 异常语义
- 对外边界 DTO：写类级 KDoc，并用 `@property` 逐项说明字段含义、是否可为空、默认行为与外部语义
- 某个复杂方法的事务边界、数据流或性能取舍：写在方法前的 KDoc 或紧贴方法的说明块
- 某个局部语句的非显然决策：写短行内注释，但要克制

## 5. 触发“必须更新注释”的改动

如果本次改动改变了以下内容，必须同步检查并更新已有注释：

- 类型职责
- 事务边界
- 数据流或依赖方向
- 安全策略
- 性能策略
- 对外读取边界
- 模块当前阶段的落地策略
- 边界 DTO 的字段语义、可空规则或默认行为

旧注释和新实现冲突时，优先更新或删除旧注释，不保留“过期但先不动”的说明。

## 6. KDoc 风格的最低要求

对“必须补注释”的声明，默认满足以下要求：

- 使用 KDoc 风格 `/** ... */`
- 类型级 KDoc 说明职责、边界、不负责什么和放在当前层的原因
- 接口方法级 KDoc 说明方法作用和关键设计思路
- 对外边界 DTO 的类级 KDoc 说明用途，`@property` 说明字段作用与必要约束
- 每个非显然参数都应有 `@param`
- 非 `Unit` 返回值默认应有 `@return`
- 有明显副作用、异常语义、顺序要求或调用约束时，应在正文或 `@throws` 中说明
