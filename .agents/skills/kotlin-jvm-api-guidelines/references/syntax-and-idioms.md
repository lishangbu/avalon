# Kotlin/JVM 语法与惯用写法

## 使用范围

当你在决定 Kotlin 代码“该怎么写”时，优先看这份参考。它更关注语言结构和可读性，而不是兼容性细节。

## 1. 先区分内部实现和公共 API

- 内部实现优先追求清晰、直接和低样板。
- 公共 API 额外要求显式性，尤其是 visibility、返回类型、nullability 和错误语义。
- 如果一个类型会跨模块复用，先按公共 API 标准设计，再回头看是否需要收紧可见性。

## 2. 优先做小而稳的核心概念

- 先定义最小核心概念，再在其上添加便利扩展。
- 如果一组能力都建立在同一个核心操作上，优先把核心操作做清楚，而不是堆很多平行入口。
- 复用 Kotlin 已有类型时，优先复用；只有在领域语义明显更强时，才引入新抽象。

## 3. 成员、顶层函数与扩展函数怎么选

- 类型本身的核心状态和不变量，优先放成员。
- 如果所有状态都能通过参数提供，优先考虑顶层函数。
- 基于类型核心概念派生出来的便利行为，优先考虑扩展函数和扩展属性。
- 计算型属性和普通方法，默认优先考虑扩展；常规属性、override 和 operator 默认优先保留为成员。

## 4. DSL、默认参数与布尔参数

- 只有在 DSL 明显提升可读性时才用 DSL，不要为了“像 Kotlin”而强行 DSL 化。
- Builder DSL 中的必填项，优先作为函数参数传入，不要全靠 lambda 内部后置设置。
- 对于有自然 happy path 的 API，可以提供默认参数；如果默认值本身有争议，往往说明抽象还没收敛。
- 公共 API 中尽量避免布尔参数；优先拆成命名更明确的函数，或改成枚举 / 策略对象。

## 5. 类型选择

- `data class` 适合稳定数据载体，不要把复杂行为和隐式副作用塞进去。
- `value class` 适合轻量包装单值并强化语义，但要注意 JVM 表现和装箱行为；只有明确存在 Java 调用方时，再额外考虑 Java 互操作。
- 对闭合层级优先考虑 `sealed class` / `sealed interface`，尤其是状态机、结果集和受限层级。
- 公共 API 默认优先接收和返回只读集合；不要把可变集合和数组直接暴露成常规 API 形态。

## 6. nullability 与错误语义

- “可能没有值”优先用 nullable return。
- “调用失败”优先用异常或 `Result`，不要把真实失败也混进 `null`。
- 同一组 API 要保持同一种错误处理风格；如果同时提供抛异常版和捕获版，命名要清晰，例如 `foo` / `fooCatching`。
- 不要把异常当作正常控制流。

## 7. 文件与命名

- 文件名描述内容，不要使用 `Util` 这类空洞命名。
- 顶层声明较多时，文件名要能反映这批声明的职责。
- JVM 项目里要注意顶层声明生成的 file facade，避免未来出现重复 JVM 类名。

## 8. 公共 API 的显式性

- 公共 API 优先显式写 visibility。
- 公共函数和属性优先显式写返回类型，避免因类型推断变化而意外改变 API 面。
- 对库或共享模块，优先考虑启用 explicit API mode。
