# 注释回补策略

给现有代码回补注释时，优先按下面的顺序做，不要一上来全仓机械铺满：

1. `shared-kernel` 下对外公开的核心抽象
2. 接口、仓储契约、应用服务契约和跨上下文公共契约的方法级 KDoc
3. 对外 Request/Response、事件 payload、ACL、快照类型的类级 KDoc 与 `@property`
4. 认证、会话、令牌、权限等安全敏感路径
5. 复杂 `application service`
6. 复杂 SQL-first repository
7. `outbox`、幂等、分发、批处理、递归装配
8. 模块入口文件

## 回补原则

- 优先在当前改动直接触及的文件里顺手补齐
- 优先补最容易误解、最影响维护和最容易引发安全或一致性误判的地方
- 接口和公共契约优先回补类型级 KDoc，再回补方法级 `@param` / `@return`
- 对外 DTO 优先先补类级用途说明，再补最容易误解的字段 `@property`
- 不要为了“覆盖率好看”给简单 DTO 或直白 CRUD 机械加注释
- 如果旧注释已经过时，先修正旧注释，再新增别的说明
- 如果一个文件已经靠命名和结构说清楚了大部分逻辑，只补最关键的那一小段
